# Database Migration Runbook

## Overview

Database migrations are managed by [Alembic](https://alembic.sqlalchemy.org/) with SQLAlchemy ORM models defined in `src/db/models.py`. The migration scripts live in `alembic/versions/`.

## Common Commands

### Apply All Pending Migrations

```bash
make migrate
# or directly:
alembic upgrade head
```

### Check Current Migration State

```bash
alembic current
```

### View Migration History

```bash
alembic history --verbose
```

### Create a New Migration

```bash
make migration
# Prompts for a message, then runs:
# alembic revision --autogenerate -m "your message"
```

Always review the generated migration file before applying. Autogenerate detects:
- New tables/columns
- Dropped tables/columns
- Changed column types
- New indexes/constraints

It does NOT detect:
- Table/column renames (appears as drop + create)
- Changes to column data
- Custom SQL operations

### Rollback Last Migration

```bash
alembic downgrade -1
```

### Rollback to Specific Revision

```bash
alembic downgrade <revision_id>
```

### Stamp Database Without Running Migrations

Use when the database was created by `Base.metadata.create_all()` (before Alembic was set up) and you want Alembic to recognize the current state:

```bash
alembic stamp head
```

## Migration Best Practices

1. **Always review autogenerated migrations** — Check that `upgrade()` and `downgrade()` are correct
2. **Test migrations locally** — Run `upgrade` then `downgrade` then `upgrade` again
3. **Never edit applied migrations** — Create a new migration instead
4. **Include both upgrade and downgrade** — Every migration must be reversible
5. **Back up before production migrations** — `pg_dump -U postgres arbitrage_bot > backup.sql`

## Initial Schema

The initial migration (`001_initial_schema.py`) creates 8 tables:

| Table | Purpose |
|-------|---------|
| `chains` | Supported blockchain networks |
| `dexes` | DEX configurations per chain |
| `tokens` | Token addresses and metadata |
| `opportunities` | Detected arbitrage opportunities |
| `transactions` | Submitted transactions |
| `trade_results` | Execution results and P&L |
| `liquidation_opportunities` | Detected liquidation targets |
| `execution_log` | Audit log of all actions |

The initial migration checks `_table_exists()` before creating — safe to run on databases that already have tables from `create_all()`.

## Troubleshooting

### "Target database is not up to date"

The database has unapplied migrations. Run:

```bash
alembic upgrade head
```

### "Can't locate revision"

The `alembic_version` table references a revision that doesn't exist in the `alembic/versions/` directory. Reset:

```bash
alembic stamp head
```

### "Table already exists"

The initial migration handles this with `_table_exists()` checks. If a later migration hits this, the migration needs a conditional check:

```python
from alembic import op
import sqlalchemy as sa

def upgrade():
    conn = op.get_bind()
    if not conn.dialect.has_table(conn, "table_name"):
        op.create_table(...)
```

### Migration on Production

```bash
# 1. Back up database
pg_dump -U postgres arbitrage_bot > backup_$(date +%Y%m%d_%H%M%S).sql

# 2. Stop the bot (prevents writes during migration)
docker-compose stop arb-bot liquidation-bot

# 3. Apply migrations
alembic upgrade head

# 4. Verify
alembic current

# 5. Restart bot
docker-compose start arb-bot
```
